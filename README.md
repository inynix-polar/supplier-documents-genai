# Тестовые задания (vibe-coding, ~1 час)

Часовые тестовые задания для кандидатов. Каждая задача лежит в своей папке как
**самодостаточная заготовка** — небольшой изолированный проект на тему обработки и
анализа технических документов поставщиков. Доступ к каким-либо внутренним
репозиториям не требуется, всё запускается локально.

## Общие правила

- **Формат**: vibe-coding — LLM-ассистент (Claude Code, Cursor и т.п.) разрешён и
  приветствуется. Оцениваем не память синтаксиса, а декомпозицию, проверку результата
  за моделью и инженерные решения.
- **Тайм-бокс**: ~60 минут. Не успел всё — нормально; в `NOTES.md` опиши, что сделал бы
  дальше и какие были компромиссы.
- **Зависимости**: каждая папка изолирована, поднимается по своему README. Тяжёлую
  инфраструктуру (Postgres/Redis/MinIO/живой LLM) поднимать не нужно — используются
  sqlite, LLM-клиенты в режиме `LLM_FAKE=1` и синтетические данные.
- **Соглашения**: код/комментарии — на русском (термины и имена переменных — английский
  snake_case). Pydantic для данных, бизнес-правила — в конфиги, а не в хардкод.
- **Git (обязательно)**: работай в git, делай **осмысленные коммиты в стиле
  [Conventional Commits](https://www.conventionalcommits.org/)** (`feat:`, `fix:`,
  `test:`, `refactor:`, `docs:`, `chore:` …). Один гигантский коммит «всё сразу» —
  минус. История должна читаться как ход твоей работы.
- **`NOTES.md` (обязательно)**: в каждой папке есть шаблон `NOTES.md` — заполни его
  (решения, допущения, что не успел, как проверял).

## Что оцениваем у всех

1. **Декомпозиция** — разбил ли задачу на понятные шаги.
2. **Качество за LLM** — заметил ли галлюцинацию/багу, сгенерированную моделью.
3. **Интеграция** — переиспользовал ли данные абстракции (контракты, клиенты, реестры),
   а не написал «сбоку».
4. **Тесты/проверяемость** — есть ли минимальный тест или скрипт-демо.
5. **Коммуникация** — внятный `NOTES.md` с решениями и допущениями.

## Бонусы (по желанию, плюс к оценке)

Не обязательны для зачёта, но ценятся и есть в каждом задании:

- **Базовый CI/CD** — пайплайн (например GitHub Actions), который на push/PR хотя бы
  прогоняет линтер и тесты.
- **Docker** — `Dockerfile` (и при желании `docker-compose.yml`), поднимающий решение
  одной командой.
- **Управление зависимостями** — `requirements.txt` оставлен как стартовая точка;
  плюсом будет перевод на **Poetry** (`pyproject.toml`).
- **Нормальные тесты** — не только happy-path: пограничные случаи, фикстуры, изоляция.

## Задания по ролям

| Роль | Папка | Суть |
|------|-------|------|
| Python Middle Backend | [`backend/`](backend/README.md) | Идемпотентная загрузка архивов по sha256 + Alembic-миграция + `GET /api/stats` |
| Python/TS (React) Full-stack | [`fullstack/`](fullstack/README.md) | UI «Regex Lab»: подбор regex по примеру/анти-примеру через LLM + живая подсветка |
| Middle Data Engineer | [`data/`](data/README.md) | ETL по structlog-логам → DuckDB + метрики пайплайна (синтетика в комплекте) |
| Middle ML Engineer | [`ml/`](ml/README.md) | Эмбеддинг-матчер атрибутов вместо fuzzy + offline-eval (P/R/F1) |
| Middle Gen-AI Engineer | [`genai/`](genai/README.md) | Извлечение атрибута: structured output + few-shot + грунтинг (анти-галлюцинация) |

Каждая папка содержит свой README с задачей, заготовки кода с пометками
`TODO(кандидат)`, данные/фикстуры и инструкцию запуска.
