## Решения и допущения

### Контракты и конфигурация

- Внешний `LLMExtractionResponse` отделён от итогового `ExtractedValue`: первый запрещает
  лишние поля и строго валидирует `confidence=high|medium|low`, второй хранит результат
  grounding и `rejected_reason`.
- `value=null` — нормальный ответ «не найдено»: тогда `unit/source_quote=null`, а
  `confidence=low`. Пустые строки не считаются отсутствием значения.
- Baseline оставлен permissive и не использует строгую LLM-схему, иначе контрольная
  стратегия искусственно улучшилась бы или падала на нештатном `confidence`.
- `max_attempts`, дополнительная проверка value-in-quote и параметры eval находятся в
  Pydantic-конфигах. Тип, единица, максимальная длина и OCR-алиасы задаются в реестре
  атрибутов, а не размазаны по extractor.
- Canonical unit из реестра считается доверенной метаинформацией только для `unit`.
  Значение разрешено брать исключительно из документа.

### Grounding и retry

- Exact quote — безусловный инвариант: регистр и пробелы цитаты не исправляются. Отдельная
  нормализация используется только для проверки, что `value` действительно содержится в
  evidence.
- Числа сравниваются целыми токенами, поэтому `50` не подтверждается цитатой `150`.
  OCR-правила схлопывают одиночные цифры и для DN явно разрешают `O/О → 0`, но применяют
  алиасы только внутри числовой последовательности с реальной цифрой: обычные слова не
  создают искусственный `0`.
- Retry выполняется только при JSON/schema error. Второй запрос получает исходный документ,
  предыдущий ответ и типы `ValidationError`; это не слепой повтор того же запроса.
- Валидная, но отсутствующая в тексте цитата немедленно отбраковывается без повторного
  LLM-вызова: повтор здесь увеличил бы стоимость и маскировал галлюцинацию.
- Неожиданные исключения клиента не проглатываются. После исчерпания schema retry результат
  явно помечается `invalid_model_response`.
- `confidence` сохраняется для аудита, но не используется как порог: без калибровки такой
  порог создавал бы ложное ощущение безопасности.

### Методика eval

- Для каждого `case.id` вычисляется стабильный seed. Baseline и grounded получают отдельные
  `random.Random` с одинаковым seed, поэтому их первый fake-ответ парный и сравнение честное.
- Accuracy считает совпадение после доменной нормализации; `null == null` считается верным.
- Hallucination rate в отчёте — `grounding_violations / answered`: ненулевые ответы без
  точной и содержательной цитаты. OCR-значение может отсутствовать буквально в тексте,
  поэтому проверять только `value in text` некорректно.
- False-positive rate считается отдельно на кейсах `expected=null`. Это важно: цитата может
  быть реальной, но относиться к другому атрибуту.
- Unit accuracy считается только среди уже правильно извлечённых ненулевых значений, чтобы
  не награждать неверный `999` за случайно совпавшую единицу.
- Абсолютный порог accuracy не задавался: fake-клиент намеренно примитивен и не понимает
  семантику материалов/конфликтов. Acceptance-инварианты: у baseline есть violations,
  grounded имеет 0 violations и не уступает baseline по accuracy/selective accuracy.

## Результаты eval

Команда: `LLM_FAKE=1 uv run --frozen python -m eval.run_eval --seed 42`.
Два последовательных запуска дали идентичный полный отчёт.

| Стратегия | Accuracy | Hallucinations / answered | False positives | Coverage | Selective accuracy | Rejections |
|---|---:|---:|---:|---:|---:|---:|
| baseline | 2/16 (12,5%) | 8/16 (50,0%) | 3/3 | 16/16 (100,0%) | 12,5% | 0/16 |
| grounded | 3/16 (18,8%) | 0/8 (0,0%) | 2/3 | 8/16 (50,0%) | 25,0% | 8/16 |

Grounded устранил все неподтверждённые ответы и улучшил accuracy на 6,2 п.п., но не решил
семантический выбор: год выпуска или реальное число другого атрибута могут иметь корректную
цитату и всё равно быть неверным ответом. Grounding — необходимая, но не достаточная защита.

## Стоимость, латентность и batching

- Нормальный путь — один LLM-вызов; JSON/schema error — не более двух. В seed-42 eval
  среднее равно `1.00` для обеих стратегий, потому что fake всегда возвращает валидный JSON.
- Few-shot увеличивает входные токены, но снижает неоднозначность схемы. В production стоит
  измерять token usage и p50/p95 latency на реальном endpoint.
- Сейчас один атрибут извлекается одним вызовом. Батч нескольких атрибутов уменьшит общую
  стоимость и latency, но усложнит схему, retry и изоляцию ошибок. Практичный следующий шаг —
  небольшие батчи с evidence для каждого атрибута, а не один ответ на весь документ.

## Native structured outputs и совместимость

- Реализация использует строгий JSON + Pydantic и поэтому работает со старым
  OpenAI-compatible chat-completions endpoint.
- Native structured outputs/tool-calling в production уменьшили бы число parser retry, но
  требуют проверки capabilities конкретного провайдера. Текущий parser остаётся graceful
  fallback и принимает только чистый JSON либо целиком fenced JSON.

## Что не сделано / сделал бы дальше

- Не подключался живой LLM: это прямо не требуется заданием и сделало бы CI зависимым от
  сети, ключей и стоимости.
- Не добавлен Docker: приоритет отдан обязательной логике, `uv`, CI и edge-case тестам;
  проект и так поднимается одной воспроизводимой командой без сервисов.
- Нужен semantic verifier, проверяющий связь цитаты именно с выбранным атрибутом, обработку
  «старое/актуальное» и материалов разных деталей. Возможные варианты: второй constrained
  pass, span classification или доменные правила вокруг синонимов.
- Для статистических выводов нужен больший размеченный датасет и агрегация нескольких seed
  с доверительными интервалами; 16 синтетических кейсов — инженерный smoke-eval.
- В production добавил бы token/latency telemetry, rate-limit retry с backoff, provider
  capabilities для native schema и контролируемое батчирование.
